# Project Rules

## Remote Server Management

- Treat local and remote management as separate targets. Local state remains in the local CC Switch data directory and must not become the source of truth for a remote server.
- A remote server owns its own CC Switch state, database, tool config files, MCP entries, prompts, and skills. The desktop app stores only remote connection profiles and cached health metadata.
- Build remote execution around a Rust CLI helper running on the remote host. The helper should reuse the Rust service/core logic that backs the Tauri app instead of reimplementing provider, MCP, prompt, or skill logic in TypeScript.
- Keep any NPM package as an optional installer or thin wrapper for downloading and invoking the Rust binary. Do not put durable business logic in an NPM implementation.
- Communicate with remote hosts through SSH commands that return stable JSON. Avoid editing remote files directly from the GUI through ad hoc SFTP path logic.
- Keep remote feature code isolated behind a remote adapter and remote UI shell so upstream local app changes remain easy to merge.
- Mark feature parity explicitly. Provider switching, MCP, prompts, skills, import/export, and health checks should aim for parity; tray controls, desktop deeplinks, browser OAuth flows, local terminal launch, and local proxy takeover are local-only unless a remote-safe equivalent is designed.
- Do not silently merge local and remote configuration. Any import, export, or copy between targets must be explicit, previewable, and reversible.
- Do not persist remote API keys or provider secrets in the local desktop database. Secrets should stay on the remote host unless the user explicitly performs an export or migration workflow.
- Prefer small, upstream-friendly changes: add new remote modules, commands, and UI entry points before modifying existing provider, MCP, prompt, skill, or settings flows.
- Remote helper commands must be thin adapters over shared Rust core/service logic. Do not add helper-only implementations for tool version checks, install/update commands, provider import/switch logic, MCP import aggregation, prompt management, skill management, sessions, Hermes memory, or OpenClaw settings.
- Tool environment checks and lifecycle actions must share the same Rust implementation for local and remote targets. If local behavior changes for PATH search, version classification, latest-version lookup, installer fallback, or update command anchoring, the remote helper must pick it up through the shared implementation rather than copying the change.
- Secret redaction for remote provider payloads must stay centralized. If provider schemas or secret-field detection rules change, update the shared redaction/restore path and its tests rather than adding ad hoc redaction in remote commands.
- Remote UI should reuse local page components for equivalent workflows. Add remote-specific shells only for target selection, SSH/helper status, capability gating, and host management; do not rebuild provider, MCP, prompt, skill, environment, or settings controls with separate visual patterns.
- Every new remote capability must declare whether it is parity, remote-adapted, or unsupported, and include focused tests for local/remote adapter behavior. A helper capability should not be advertised until the corresponding command is implemented and covered.
- Every remote management feature must start from a local feature parity audit. Before implementing or changing a remote feature, identify the corresponding local page, component tree, hooks, service calls, validation rules, loading states, error handling, i18n keys, tests, and user workflow. The remote implementation must document and preserve the same behavior unless a remote-specific difference is explicitly justified.
- Remote management UI must reuse the local UI structure and components for equivalent workflows. This applies to providers, settings, routing, MCP, prompts, skills, sessions, Hermes memory, environment checks, import/export, authentication, and tool lifecycle pages. Do not create visually separate, simplified, or placeholder remote pages when a local page/component already exists.
- Remote management logic must reuse local business logic through target-aware adapters, shared hooks, shared Rust services, or shared command handlers. Do not add remote-only shortcuts that skip local validation, state transitions, provider mapping, secret handling, lifecycle checks, queue updates, configuration normalization, or status refresh behavior.
- A remote feature is not considered implemented just because the page renders or a button exists. It must complete the full workflow: load remote state, display the same effective state as local, validate input, save to the remote host, refresh state after mutation, show localized success/failure feedback, handle unsupported helper capabilities, and include focused tests for the local/remote behavior split.
- For any local feature migrated to remote management, keep a support matrix in code or documentation: parity, remote-adapted, or unsupported. Unsupported parts must be hidden, disabled, or clearly explained in the existing UI pattern; they must not appear as clickable controls that do nothing or fail with low-level errors.
- Remote homepage controls must preserve the same dependency logic as local controls. If a local action requires an app-specific route, provider state, installed tool, authenticated account, or confirmed setting, the remote action must enforce the equivalent remote precondition before it can run.
- Remote feature additions must include an explicit comparison against the current upstream/local implementation during review. If upstream changes a local page or workflow later, the corresponding remote feature must be rechecked and updated through shared components where possible.
- Remote command failures must not tear down a long-lived helper session unless the transport/protocol is actually unusable. Command-level errors such as tool install failures, unsupported commands, validation failures, or `ok` responses without payload data must not close or recreate the helper session because the remote routing runtime can live inside that session.
- Remote helper JSON commands that return `ok: true` without `data` must be accepted for unit-returning operations. Add regression tests for remote commands that intentionally return no payload, especially settings toggles and routing toggles.
- Any remote runtime action that can affect existing remote state must be failure-preserving. If updating settings, providers, routing, tools, MCP, prompts, skills, sessions, Hermes memory, import/export, authentication, or plugin config fails, the previous remote state and any running remote runtime must remain intact unless the user explicitly requested a destructive action or the transport was lost.
