# CC Switch Remote v3.16.6

> 这是 `cc-switch-remote` 的过渡版本号。应用版本继续沿用递增的 `3.16.x`，以保证 Tauri 自动更新能够从已发布的 `v3.16.5` 正常升级。
>
> **上游基线**：CC Switch `v3.16.3`
>
> **远程版构建**：`cc-switch-remote v3.16.6`

---

## 概览

本版主要完成与上游 CC Switch `v3.16.3` 的同步，同时保留 `cc-switch-remote` 的远程管理分支定位：本地管理与远程管理继续作为两个独立目标，远程主机通过纯 CLI Helper 执行管理能力，桌面端只保存连接配置和健康状态。

上游 `v3.16.3` 的重点是用量计费准确性、供应商健康检查语义、Codex 统一会话历史、关于页性能、Codex 升级修复，以及一批代理和平台兼容性修复。本版已经把这些改动合并到远程版主线，并对远程管理相关代码做了冲突处理。

---

## 版本号说明

当前远程版已经发布过 `v3.16.5`。如果回退到 `v3.16.3-remote.x`，已安装用户无法通过 Tauri updater 自动升级。因此本轮继续使用 `v3.16.6`。

后续 release note 会明确写出：

- **应用版本**：`cc-switch-remote v3.16.x`
- **上游基线**：`cc-switch v3.16.3`
- **Helper 构建**：以 release asset / commit build 标识为准

这样既保持自动更新链路可用，也避免把远程版的 patch 号误解为上游版本。

## 上游同步内容

### 供应商健康检查更轻量

供应商健康检查从“发送真实流式模型请求”调整为“HTTP 可达性探测”。这能减少第三方供应商因为 WAF、鉴权策略或流式接口差异导致的误报。

注意：可达不等于真实请求一定成功。真实路由失败和故障转移仍以实际代理流量为准。

### Codex 统一会话历史

合并上游的 Codex 统一会话历史设置和迁移/还原逻辑。开启后，官方 Codex 会话与第三方会话可以共享 resume 历史桶，减少“切换供应商后看不到会话”的情况。

### Codex 升级修复

合并上游对 Codex npm 安装升级的修复：不再优先走容易损坏平台二进制的 self-update 路径，而是通过可运行性检测和重装修复缺失的 optional dependency。

### 用量统计与代理修复

合并上游关于真实上游模型计费、格式转换路径缓存 token 统计、Claude Code Workflow 子 agent 用量、Codex `/responses` 图像整流、错标 SSE 响应聚合、Hermes YAML 重复键等修复。

### 关于页加载优化

合并上游关于页工具版本检测缓存和渐进加载优化。应用版本和工具版本不再互相阻塞，工具检测结果会在应用会话内缓存。

## 远程版处理

### 保留远程管理分支定位

本轮合并没有把远程管理逻辑混入本地状态。远程主机仍然拥有自己的 CC Switch 状态、数据库、工具配置、MCP、Prompts、Skills 和 Helper。

### 保留远程 Helper 架构

远程 Helper 继续使用纯 CLI 构建，不依赖桌面 GTK/WebKit 运行库。`remote-helper-latest` 会由单独的 Remote Helper Artifacts 工作流发布。

### 保留远程 target/adapter

供应商、设置、工具环境、路由相关能力继续通过本地/远程 target adapter 区分执行目标，避免本地 JSON 或本地数据库误写入远程状态。

### Release Note 发布流程修复

发布工作流现在会优先读取 `docs/release-notes/<tag>.md` 或 `docs/release-notes/<tag>-zh.md` 作为 GitHub Release 正文。以后 release 不再只有下载说明。

## 升级提醒

- 已安装 `v3.16.5` 的用户可以通过应用内更新升级到本版。
- 远程 Helper 与桌面应用分开发版。更新桌面应用后，建议在远程主机健康检查中确认 Helper 是否为最新构建。
- 如果使用远程路由、远程供应商或远程工具管理，请在升级后先对目标主机执行一次健康检查。
- macOS 构建当前未进行 Apple Developer ID 签名和公证，首次打开可能需要在系统设置中手动允许。
